# 基础测试流程：按游戏类型的流程预期（执行参照）

本文档供 **MCP 调用方、自动化代理与人工** 在生成或扩写 `design_game_basic_test_flow` 产物时作**思考参照**，不是对具体项目的硬性契约。实际工程接入方式、完成度、可观测接口各不相同，应始终以 **`index.json` / `06-operational-profile.md` / `05-flow-candidate-catalog.md`** 中的**可验证证据**为准。

## 1. 共同原则（所有类型）

1. **先阶段、后玩法**：若存在 `change_scene` 等场景切换，菜单阶段与关卡阶段不得混在同一步序；见 `06-operational-profile.md`。
2. **快**：基础流程步数有限时，优先覆盖「能证明游戏已处于可玩态」的一条路径，而不是穷举功能。
3. **全**：在步数预算内，尽量包含：**进入可玩态**、**至少一次核心交互（或等价 UI 操作）**、**至少一条可观察断言**（界面状态、逻辑 `check`、或快照）。
4. **可验证**：每一步应对应真实世界可复述动作；禁止无含义占位步。
5. **证据不足则阻塞或保守**：静态分析显示控件不可点、无断言候选时，应返回结构化阻塞信息或生成保守冒烟，而不是虚构目标。

## 2. 类型对照表（流程预期）

下表中的「典型步骤意图」应与 `flow_candidates` 中的 `action_candidates` / `assertion_candidates` 对齐；若项目中不存在对应证据，跳过该意图并记录在 `generation_evidence` 或人工备注中。

| 游戏类型（静态标签/关键词） | 进入可玩态后，希望快速覆盖什么 | 典型可观察点（举例） |
|----------------------------|--------------------------------|----------------------|
| **FPS / 第一人称探索** | 移动或视角相关输入一次；若仅有菜单，则进入关卡后再做一次最小交互 | HUD 可见、准星/武器 UI、`logic_state` 与场景阶段一致 |
| **平台跳跃 / 横版** | 进入第一 playable 关卡；一次跳跃或水平移动（若自动化仅支持 UI，则点到「开始/继续」并断言关卡 UI） | 生命/分数 UI、关卡内控件出现 |
| **RTS / 塔防 / 建造** | 进入战斗或沙盘；一次「选择单位/放置塔/开始波次」类操作（有则点选，无则点主界面「开始战斗」） | 资源条、波次/单位列表、暂停菜单 |
| **回合制 / 战棋 / 卡牌** | 进入对局；结束回合、出牌或「开始对战」之一 | 回合指示、手牌区、结束回合按钮状态 |
| **解谜 / 叙事 / 点击冒险** | 进入首个互动场景；一次与谜题或对话相关的点击 | 对话行、物品栏、场景状态变化（`check`） |
| **竞速 / 体育** | 进入比赛或训练；开始赛事；若仅能菜单，则「开始比赛」+ 暂停/设置其一 | 计时器、排行榜、暂停菜单 |
| **休闲 / 超休闲** | 单局极短则：开始 → 一局内最小操作 → 结算或重开 | 分数、重玩按钮、广告/继续（若有） |
| **纯 UI / 菜单驱动** | 主导航各主入口各点一次不现实时，选**主路径**：设置 → 返回 或 商店 → 返回，之一 | 面板显隐、返回后主界面一致 |
| **开放世界 / RPG（重任务）** | 往往步数不足；优先：**新游戏/继续** → 进入可操作角色状态 → 打开地图或背包一次（有证据才写） | 任务日志、小地图、角色属性面板 |
| **多人 / 联网** | 在单机自动化前提下，常只能测到**大厅/登录 UI**；勿假设匹配成功 | 连接状态文案、房间列表、错误提示 |
| **沙盒 / 创造** | 创建或进入世界；一次放置/破坏/模式切换（有 UI 则点选） | 模式指示、工具栏、库存 |

## 3. 与自动工具的关系

- **`design_game_basic_test_flow`**：按当前实现从候选中选取有限步；不自动遍历上表每一行。代理应在**读完** `04-flow-authoring-guide.md`（含类型速查）与本文档后，决定是否需要 **`update_game_basic_design_flow_by_current_state`**、手工编辑 flow JSON、或提高 `max_feature_checks`（上限见工具说明）。
- **`operational_profile` 中的 `gameplay_understanding`**：可与上表对照，用于判断「本项目更接近哪类」以优先选步。
- **存读档**：仅当脚本与 UI 证据同时存在时，自动生成存读档步；参见 flow 内 `notes` 与 `generation_evidence.save_load`。
