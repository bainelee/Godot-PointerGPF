# MCP 游戏卡死与「自动排错闭环」期望落差 — 根因说明与改造计划

> **For agentic workers:** REQUIRED SUB-SKILL: 实现阶段请使用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans，按任务逐项执行。任务使用 `- [ ]` 勾选。

**Goal:** 用普通人能读懂的方式说明：为什么在「游戏报错/卡死」时，当前 MCP 行为会像「和你一起卡住」，而不是「发现错误 → 关掉运行 → 看清错误 → 改到能跑通」的自动闭环；并给出可验收的改造方向。

**Architecture:** 说明限定在现有设计的三块——Python 侧流程执行器、Godot 侧文件桥（命令/应答/诊断文件）、以及「自动修 bug」独立工具——它们**默认没有串成一条自动链**；诊断文件**并不等价于编辑器里的全部报错**。

**Tech Stack:** Python（流程调度）、Godot 4.x（编辑器内 Play + 插件）、基于磁盘的 JSON 桥接。

---

## 第一部分：你心里期待的行为 vs 系统实际在做什么

### 你期待的大致流程（产品直觉）

1. 测试一跑，游戏里有报错或界面不动。  
2. 外面的 MCP（或助手）**马上**知道出事了。  
3. **关掉**这次运行（至少退出 Play，别让窗口一直占着）。  
4. 把**可读的错误信息**拿出来给人或给模型。  
5. **自动改代码/场景**，再跑，直到流程能走完。

### 系统实际默认在做什么

1. **每一步**都是：往项目里的一个「命令文件」写 JSON，然后**死等**「应答文件」出现且序号对上。  
2. 等的时候，Python 会**轮询**另一个「诊断快照文件」：只有里面标记成严重错误级别时，才会**提前**结束等待；否则会一直等到**这一步的超时时间**（常见几十秒）用完。  
3. **没有任何一步**会在超时或卡死时去**强制结束 Godot 进程**；最多再写一条「请停止 Play」的请求给编辑器插件，若游戏主循环已经卡死，这条请求也常常**得不到处理**。  
4. **「自动修 bug」**在代码里是一个**单独的工具**；跑基础测试流程时，**不会**在失败后自动接着调用它——只在部分失败类型的返回里**用文字建议**「你可以再去调自动修」。  
5. 若你在聊天里让助手「跑完整条命令并等它结束」，助手进程会**一直阻塞到**超时或异常返回，这段时间看起来就像「助手也卡死了」。

**一句话：**当前实现是「**按步骤约稿、等回信**」的自动化，不是「**全程托管游戏进程 + 自动迭代修复**」的自动化。

## 第二部分：造成「游戏卡死 = 外面也像卡死」的具体原因（按重要性）

### 原因 A：等待逻辑本质是「最长等到超时」，不是「一有异常就停」

- 每一步都有上限等待时间。游戏若**不回写应答**（卡死、断在主线程、弹窗挡住、调试器暂停），Python 就会**一直等到该步超时**。  
- 这段时间里，若你让助手同步执行同一条长命令，**助手界面也会一直等**，看起来像双边卡死。

### 原因 B：「诊断快照」主要来自**桥接命令的结果**，不是 Godot 里所有红字报错

- 当前示例里，诊断文件里的「严重」级别，和「某次桥接命令返回了失败」强相关。  
- **脚本解析错误、场景加载失败、引擎在输出窗口里刷的栈**，若**没有**被专门写进这份诊断文件，Python 侧的「快失败」**就触发不了**，仍然会走「等应答或等超时」。  
- 文档里说的「并行看诊断」在**契约意图**上是对的，但**实现覆盖**还没有做到「等价于人类盯着 Godot 输出窗口」。

### 原因 C：收尾「停止 Play」依赖游戏/编辑器还能处理文件命令

- 流程失败或超时后，Python 会尝试发「关闭本次运行」的请求（仍走文件桥）。  
- 若运行循环已经卡死，**读命令、写应答**的那套逻辑可能**根本跑不到**，收尾只能超时结束；**编辑器窗口和进程通常还在**。  
- 代码里**没有**「按进程号杀掉 Godot」之类的兜底，所以**从操作系统层面**并不会帮你强制清场。

### 原因 D：「自动修」不是流程的内置下一环

- 有的失败类型会在返回 JSON 里带上「建议下一步用自动修工具」的提示；**超时**类失败里**没有**统一带上同样的建议。  
- 无论如何，**没有任何调度器**在失败后自动再调自动修、再重跑全流程——那需要**外层**（例如 CI、或聊天里的助手）自己决定下一步。  
- 所以你感觉「没有按核心设计意图运作」，很大程度上是 **「意图写在文档/想象里，没有落成默认代码路径」**。

### 原因 E：你手动终止时，状态可能更乱

- 若在超时到达前强行停掉终端或中断助手，**不保证**已写入 `close` 请求或已删临时命令文件。  
- 下一次再跑可能出现「旧命令残留」「停止 Play 的标志文件残留」等，表现为更难预测（示例工程已对**停止 Play 标志**做过防误触处理，但**通用原则**仍是：异常中断越多，磁盘桥接状态越要自己留意）。

## 第三部分：与「审查 MCP 代码」结论的对照（给工程师的索引，非必读）

若需要对照实现，可按下面映射到仓库（此处仅索引，正文不依赖路径记忆）：

| 现象 | 主要逻辑所在 |
|------|----------------|
| 等应答、轮询诊断、超时 | 流程执行模块中的「等待应答」循环 |
| 诊断严重级别从哪来 | Godot 侧「诊断写入」仅在桥接处理结果上标记失败时为高严重级别 |
| 超时/步骤失败后的收尾 | 服务入口在捕获异常后调用「请求关闭项目」的辅助函数 |
| 启动编辑器、不杀进程 | 服务入口用「拉起编辑器进程」的辅助函数，分离启动，无对应「保证杀死」 |
| 自动修 | 单独工具入口，由「跑基础流程」在**部分**错误返回中建议调用，非自动链接 |
