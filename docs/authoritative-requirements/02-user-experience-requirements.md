## 体验清单

### UX-001（默认自动修复，最少打扰）

- 体验编号：UX-001
- 用户角色：提交问题的项目使用者
- 使用目标：用户只需描述问题，系统自动完成修复并给出结果。
- 用户操作路径：
  - 用户输入自然语言问题描述。
  - 系统自动进入修复流程并持续执行。
  - 系统在完成后一次性返回结果，或在无法继续时请求用户介入。
- 期望反馈（界面/日志/提示）：
  - 过程应有可追踪日志，但默认不频繁打断用户。
  - 结果反馈应包含：是否修复成功、修复内容、复测结果。
  - 若无法完成，应明确说明原因和下一步需要用户提供的信息。
- 不可接受体验：
  - 刚开始处理就频繁询问用户细节。
  - 没有进行实际运行测试就宣称修复完成。
  - 没有复测证据。
- 可接受耗时范围：
  - 常规问题：尽量在一次自动处理周期内完成。
  - 复杂问题：允许多次循环，但必须在达到时间阈值后主动告知无法完成。
- 验收方式（人工/自动化）：
  - 自动化检查：验证是否执行了“验证 -> 修复 -> 复测”流程。
  - 人工检查：确认用户在非异常情况下未被频繁打扰。
- 备注：默认策略是“自动执行优先，异常再打扰”。

### UX-002（基础测试流程反馈应直接体现两类目标）

- 体验编号：UX-002
- 用户角色：执行基础流程验证的项目使用者
- 使用目标：用户用一句命令即可知道“工具是否正常工作”和“游戏流程是否正常运行”。
- 用户操作路径：
  - 用户输入：`设计一个基础测试流程` 或 `跑一遍基础测试流程`。
  - 系统自动执行，不要求用户逐步确认。
  - 系统返回结构化结果，明确两类结论和证据。
- 期望反馈（界面/日志/提示）：
  - 在 shell 中持续输出步骤级行为播报，文本可读、语义清楚。
  - 最终结果必须分开显示：
    - 工具执行能力是否通过。
    - 游戏流程运行能力是否通过。
  - 若未通过，直接给出失败步骤、原因和下一步动作。
- 不可接受体验：
  - 只返回“命令成功”但没有流程级证据。
  - 播报内容难以阅读，存在乱码或无意义输出。
  - 把“流程文件生成成功”误当成“基础流程测试通过”。
- 可接受耗时范围：
  - 基础流程应在合理时长内完成整段游戏路径验证。
  - 若超过时限，应明确提示超时并给出当前进度。
- 验收方式（人工/自动化）：
  - 自动化检查：校验结果中是否同时存在两类结论字段。
  - 人工检查：抽查 shell 播报是否可读、是否覆盖关键步骤。
- 备注：本条强调“结果可读、结论可分、证据可追溯”。
