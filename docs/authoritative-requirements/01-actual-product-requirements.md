## 需求清单

### R-001（自动化问题修复闭环）

- 需求编号：R-001
- 需求标题：基于用户问题描述的自动化修复
- 背景与目的：用户使用 `GPF` 的核心目的，是让系统自动修复已发现的问题，尽量减少人工介入。
- 输入：用户自然语言问题描述（例如：`这个按钮无法点击`）。
- 输出：可复现问题的测试流程、问题原因说明、修复结果、复测通过证据。
- 约束条件：
  - 接到问题后，`MCP` 必须自行设计并执行测试流程。
  - 必须进行实际游戏运行测试，不能只做静态分析或只生成流程文件。
  - 必须循环执行以下步骤：验证问题 -> 找到问题出现的原因 -> 修复问题 -> 再次运行游戏测试。
  - 只有在以下情况才允许中断并打扰用户：
    - 出现无法结束的循环。
    - 超过设定时间仍无法修复。
- 验收标准（可执行、可量化）：
  - 对于给定问题描述，系统能自动生成并执行至少 1 条可运行测试流程。
  - 过程日志中包含“验证、定位、修复、复测”四类记录，且至少完成 1 个完整循环。
  - 最终状态必须明确为：`修复成功` 或 `超时/无法完成并已通知用户`。
  - 若修复成功，必须有至少 1 次复测通过证据。
- 优先级（P0/P1/P2）：P0
- 备注：该需求是产品核心能力，后续开发与验证默认优先满足此条。

### R-002（基础测试流程命令的双重目标）

- 需求编号：R-002
- 需求标题：基础测试流程命令同时服务“工具可用性验证”和“游戏流程可运行性验证”
- 背景与目的：当用户说“设计一个基础测试流程”或“跑一遍基础测试流程”时，通常是在验证两类目标，且可能同时存在。
- 输入：用户自然语言命令（例如：`设计一个基础测试流程`、`跑一遍基础测试流程`）。
- 输出：
  - 一份可执行的基础测试流程（不是仅生成文件）。
  - 一组可审计的执行证据（工具行为与游戏行为）。
- 约束条件：
  - 必须覆盖目标 A：验证 `GPF` 是否正确运作，包括：
    - 能自动设计测试流程。
    - 能自动执行测试流程。
    - 能在游戏内执行操作。
    - 能在 shell 中实时播报行为，且播报文本清晰可读。
  - shell 播报格式必须固定为每个阶段两行：
    - 第 1 行：`[GPF-FLOW-TS] YYYY-MM-DD T HH:MM:SS`，使用本地系统时间。
    - 第 2 行：中文语义行，仅输出用户可读信息。
  - shell 播报不得显示技术字段行（例如 `run=`、`phase=`、`id=`、`action=`、`bridge_ok=`、`verified=`）。
  - 三阶段中文语义必须明确分别显示：
    - `开始执行:...`
    - `执行结果:...(通过/失败)`
    - `验证结论:通过/失败-目标:...`
  - 一次测试流程结束后（通过/失败/超时/门禁失败），必须关闭本次测试项目会话，不允许保留运行中的测试项目。
  - 必须覆盖目标 B：验证游戏在大规模调整后是否仍可正确运行。
  - 基础测试流程必须包含“有一定游戏流程”的步骤，不允许退化成只检查命令退出码的最小可用检查。
- 验收标准（可执行、可量化）：
  - 每次基础测试流程执行后，必须输出两类结论：
    - 工具可用性结论（目标 A 是否通过）。
    - 游戏流程可运行性结论（目标 B 是否通过）。
  - 执行日志中必须存在步骤级播报记录，且记录内容可读，不是乱码或原始代码片段。
  - 抽查任一步骤时，必须能看到本地时间戳行 + 中文语义行；不得看到技术字段行。
  - 执行结果中必须包含项目关闭证据（`project_close.requested=true`）。
  - 流程步骤数和步骤类型应体现“进入游戏、操作、状态检查、结果确认”的完整路径。
- 优先级（P0/P1/P2）：P0
- 备注：后续所有“基础测试流程”相关功能设计与验证，默认按此需求解释用户意图。
